# Alfa Only Assistant — MVP Foundations (Stage 1)

Этот репозиторий содержит **первый этап** учебного проекта AI-ассистента для персонального менеджера Alfa Only.

## Что сделано в Stage 1 (без AI)

На этом этапе реализован базовый backend-контур на Python:
- каркас сервиса на FastAPI;
- статический frontend для демонстрации MVP без Swagger;
- доменные модели (клиенты, коммуникации, продукты, CRM-заметки, follow-up);
- SQLite-хранилище;
- отдельный reset/seed-скрипт для MVP-данных;
- базовые API для списка клиентов, карточки клиента, списка коммуникаций и создания CRM-заметки;
- endpoint для сбора feedback менеджера;
- OpenAPI-совместимый контракт как основа для следующего этапа.

> Важно: AI-компоненты (генерация скриптов, саммаризация, приоритизация через LLM и т.п.) в Stage 1 **не реализуются** по условию.

## Этапы проекта

### Stage 1 — Foundation (реализован)
- Декомпозиция домена и API-контрактов.
- Подготовка мок-данных, локальной SQLite БД и seed-скрипта.
- Реализация базовых non-AI endpoint'ов.
- Логирование действий менеджера (feedback).

### Stage 2 — Business Logic (следующий)
- Rule-based приоритизация диалогов и next-best-action без LLM.
- Расширение explainability, фильтров и бизнес-правил.

### Stage 3 — AI Integration (позже, по запросу)
- Подключение LLM через Groq gateway.
- Генерация скриптов, саммари, режим возражений.
- Explainability и anti-hallucination guardrails.

### Stage 4 — Demo Readiness
- UX-полировка, метрики, сценарии демо и стабильность.

## Новый UI-концепт

Текущий frontend перестроен под сценарий "сначала диалоги, потом действия":
- главная вкладка — лента диалогов в стиле мессенджера;
- каждая запись показывает mock priority score, имя клиента, mini-summary и превью последнего сообщения;
- слева от диалога отображается числовой приоритет;
- по клику на приоритет открывается модальное окно с формулой и весами;
- список задач временно убран из MVP, чтобы не размывать основной сценарий.

Актуальная UX-структура:
- пока диалог не выбран, центральный экран полностью отдан под список диалогов;
- после выбора диалога слева остается только компактный мини-бар с аватарками клиентов;
- по центру открывается сам чат в стиле мессенджера;
- справа закреплены priority score, контекст клиента и отдельное AI assistant окно;
- верхнее меню списка диалогов скрывается в открытом chat-mode;
- блок фиксации результата из интерфейса убран;
- блоки `CRM notes` и `follow-up` из текущего UI убраны, чтобы не перегружать основной сценарий.

Правое AI-окно:
- это отдельный UI-блок для общения менеджера с AI assistant;
- в будущем assistant будет видеть открытый диалог и на его основе предлагать персонализированные скрипты продаж;
- пока окно реализовано только как UX-заглушка без AI-логики, без моделей и без backend-генерации;
- текущая цель этого блока — заранее зафиксировать место в интерфейсе и пользовательский сценарий.

Важно:
- AI-функции пока не реализованы;
- mini-summary и priority score сейчас используются как UI/mocked contract для дальнейших этапов;
- реальный расчет приоритета и генерация summary будут подключаться позже.

## Формула приоритета

Базовая запись:

```text
PriorityScore = w1*T_wait + w2*C_value + w3*U_comm + w4*P_sale + w5*R_churn
```

Текущая конфигурация весов для UI:

```text
PriorityScore = 0.25*T_wait + 0.30*C_value + 0.20*U_comm + 0.15*P_sale + 0.10*R_churn
```

Смысл факторов:
- `T_wait` — срочность по времени ожидания ответа.
- `C_value` — ценность клиента.
- `U_comm` — срочность и характер текущей коммуникации.
- `P_sale` — склонность к покупке / sales potential.
- `R_churn` — риск потери клиента.

В UI добавлена возможность легко подправить веса в модальном окне. Пока это UX-настройка без backend-логики.

### Формула ценности клиента

```text
C_value = 0.6*PortfolioScore + 0.4*ActivityScore
```

`PortfolioScore`:
- минимальный портфель — `0.2`
- средний премиальный клиент — `0.5`
- крупный активный клиент — `0.8`
- VIP / очень высокий потенциал — `1.0`

`ActivityScore`:
- низкая — `0.2`
- умеренная — `0.5`
- высокая — `0.8`
- очень высокая / регулярная — `1.0`

### Формула характеристики коммуникации

```text
U_comm = 0.5*UrgencyScore + 0.3*ToneScore + 0.2*ResponsivenessPattern
```

`UrgencyScore`:
- `U1` — низкая срочность — `0.2`
- `U2` — обычная рабочая коммуникация — `0.5`
- `U3` — повышенная срочность — `0.8`
- `U4` — критическая срочность — `1.0`

`ToneScore`:
- `T1` — спокойный / нейтральный — `0.2`
- `T2` — заинтересованный / вовлеченный — `0.5`
- `T3` — напряженный / нетерпеливый — `0.8`
- `T4` — негативный / раздраженный — `1.0`

`ResponsivenessPattern`:
- `R1` — низкая вовлеченность — `0.3`
- `R2` — умеренная вовлеченность — `0.5`
- `R3` — высокая вовлеченность — `0.8`
- `R4` — чувствительный к скорости ответа — `1.0`

### Формула склонности к покупке

```text
P_sale = f(product_fit, free_cash, past_response, portfolio_gap, life_event)
```

На текущем этапе эта часть только зафиксирована в README и UI-контракте. Реальную бизнес-логику будем подключать в Stage 2 и далее.
